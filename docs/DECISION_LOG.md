## Phase 3B Decisions

### D001: I18n Approach — Runtime JSON vs Build-Time (2026-02-21)

**Context:** Need to make the UI multilingual. Angular has several i18n approaches.

**Options considered:**
1. **Angular built-in i18n (`@angular/localize`)** — Compile-time, generates separate bundles per language. Great for SEO but requires separate deployments per language.
2. **ngx-translate** — Popular runtime library. Heavy dependency, may conflict with Angular versions.
3. **Custom runtime JSON + TranslatePipe** — Lightweight, no extra dependencies, loads JSON files at runtime.

**Decision:** Option 3 — Custom I18nService + TranslatePipe with runtime JSON loading.

**Rationale:**
- Zero additional dependencies (aligns with "zero ongoing costs" constraint)
- Works with single deployment on Netlify (no per-language builds)
- JSON files are easy to generate via AI batch (Phase 3C)
- TranslatePipe with impure mode updates instantly on language switch
- Simpler to understand and maintain than library abstractions

**Trade-offs:**
- No compile-time key validation (typos in `{{ 'bad.key' | translate }}` silently return the key)
- Impure pipe has higher change detection cost (mitigated by internal caching)
- No pluralization support (can add later if needed)

### D002: Individual Hadith Pages — Data Source Strategy (2026-02-21)

**Context:** Per INDIVIDUAL_HADITH_PAGES_PROPOSAL.md, three options for serving individual hadith page data.

**Options considered:**
1. **Option A: No new files** — Load full chapter JSON, render one verse. Zero storage cost.
2. **Option B: Split everything** — Each hadith its own file. Chapter page makes N requests.
3. **Option C: Hybrid** — Keep chapter files + add per-hadith stub files alongside.

**Decision:** Option C — Hybrid approach (as recommended in proposal).

**Implementation note:** The Angular VerseDetailComponent and `verse_detail` Book type are implemented. The actual per-hadith JSON files must be generated by ThaqalaynDataGenerator (Phase 3B.3/3B.4 data work). Until those files exist on the data server, navigating to a hadith detail URL will show the default "Not sure how to display" message. This is expected — the UI is ready, waiting for data.

### D003: Language Picker Placement (2026-02-21)

**Context:** Where to put the UI language selector.

**Options considered:**
1. **In header/toolbar** — Always visible, easy to find
2. **In footer** — Less prominent, follows some site conventions
3. **In settings page** — Hidden, requires navigation
4. **Floating button** — Accessible but potentially annoying

**Decision:** Option 1 — In the header, as a compact mat-select next to the site title.

**Rationale:** Language choice is a first-visit action that users need immediate access to. Header placement follows the convention of most multilingual sites (Wikipedia, Google, etc.).

### D004: RTL Implementation Scope (2026-02-21)

**Context:** How deep to go with RTL CSS support.

**Options considered:**
1. **Full CSS logical properties migration** — Replace all margin-left/right, padding-left/right with logical equivalents everywhere
2. **Targeted RTL overrides** — Add `[dir="rtl"]` rules for key layout elements only
3. **CSS framework** — Use a RTL-aware CSS framework

**Decision:** Option 2 — Targeted RTL overrides in global styles.scss.

**Rationale:** Full logical properties migration (Option 1) would touch every component SCSS file — high risk of regressions for a feature that currently only applies to Arabic UI. Targeted overrides achieve 90% of the visual correctness with minimal code changes. Can migrate to logical properties incrementally in Phase 6 code quality work.

## Phase 4 Decisions

### D009: Search Engine — Orama vs Lunr.js vs Algolia (2026-02-21)

**Context:** Full-text search across ~7,000+ hadith/verse titles and content.

**Options considered:**
1. **Algolia** — SaaS, excellent UX, free tier limited to 10k records. Requires API key and ongoing account.
2. **Lunr.js** — Client-side, popular, but no active maintenance and limited non-Latin support.
3. **Orama** — Client-side, actively maintained, built-in stemming for multiple languages, supports persistence.
4. **MiniSearch** — Client-side, lightweight, good API but less mature Arabic support.

**Decision:** Option 3 — Orama (`@orama/orama@^3.1.18`).

**Rationale:**
- Zero ongoing costs (client-side, aligns with project constraint)
- Arabic normalization built-in (diacritics stripping, alif/ta marbuta normalization)
- Supports persisted indexes (pre-built at data generation time)
- Active maintenance with good TypeScript support
- Free for any usage level

### D010: Bookmarks Storage — Dexie.js/IndexedDB vs localStorage (2026-02-21)

**Context:** Bookmarks and reading progress need persistent storage.

**Options considered:**
1. **localStorage** — Simple, synchronous, 5MB limit, no querying.
2. **Dexie.js (IndexedDB wrapper)** — Async, large storage, structured queries, table schema.
3. **Raw IndexedDB** — No dependency but verbose API.
4. **Firebase** — Cross-device sync but adds dependency and cost.

**Decision:** Option 2 — Dexie.js (`dexie@^4.3.0`).

**Rationale:**
- IndexedDB supports large datasets without the 5MB localStorage limit
- Dexie.js provides a clean Promise-based API with table definitions
- Supports export/import for user data portability
- No backend dependency (works offline, aligns with zero-cost constraint)
- Firebase can be added later as opt-in cross-device sync (Phase 6)

### D011: Audio Recitation Source — EveryAyah.com (2026-02-21)

**Context:** Need Quran audio recitation from a free, reliable CDN source.

**Options considered:**
1. **EveryAyah.com** — Free CDN, 100+ reciters, individual ayah MP3s, no API key needed.
2. **Quran.com API** — Official, but requires API key and has rate limits.
3. **Self-hosted** — Full control but massive storage (10+ GB) and bandwidth costs.

**Decision:** Option 1 — EveryAyah.com with 4 pre-configured reciters.

**Rationale:**
- Zero cost, no API key required
- Direct MP3 URLs (`https://everyayah.com/data/{subfolder}/{surah3}{ayah3}.mp3`)
- Wide reciter selection (Husary, Minshawi, Abdul Basit, Alafasy initially)
- Well-established service used by many Quran apps
- User can select reciter; more can be added easily

### D012: PWA Strategy — Service Worker + Manual Offline Downloads (2026-02-21)

**Context:** Making the app work offline for users in areas with poor connectivity.

**Decision:** Dual approach:
1. **Angular Service Worker** (`@angular/service-worker`) for automatic caching of app shell, assets, and recently visited API data
2. **Manual download manager** using IndexedDB (via OfflineStorageService) for bulk downloading Quran and Al-Kafi complete files

**Rationale:**
- Service worker handles app shell and incremental caching automatically
- Complete book files (Quran: 47MB, Al-Kafi: 84MB) are too large for automatic caching — user must opt-in
- Download progress tracking gives users confidence during large downloads
- Combines best of both: automatic for small resources, manual for bulk data

### D013: Mobile Responsive Strategy — CSS Media Queries (2026-02-21)

**Context:** Making the UI work well on mobile devices.

**Decision:** CSS media queries at 600px and 414px breakpoints, plus targeted component adjustments.

**Key changes:**
- Header search bar hidden on mobile (< 600px) — search accessible via route
- Language picker compacted (80px → 60px)
- Footer stacks vertically
- Breadcrumbs scroll horizontally
- Verse reference grid collapses from 4-column to 2-column
- Banner text scales with `clamp()` for smooth responsive sizing
