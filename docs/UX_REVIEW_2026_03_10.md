# UX Review — 2026-03-10

> **Reviewer:** Codebase-level UX review (desktop + mobile + accessibility)
> **App version:** Post-Phase 12 (all implemented phases through FB-06)
> **Production URL:** https://thaqalayn.netlify.app/
> **Method:** Systematic code review of all 30 component templates, 4 SCSS files, global styles, and i18n coverage

---

## Executive Summary

The application has matured significantly through 12 phases of improvement. Core reading features are strong: hierarchical navigation, breadcrumbs with collapse, skeleton loaders, dark mode, search with full-text and title modes, bookmarks with export/import, notes, word-by-word analysis, paragraph view, narrator hover cards, keyboard shortcuts, PWA install prompt, embedded verse support, and a mobile hamburger menu with bottom navigation. The design system (spacing scale, type scale, CSS custom properties) provides a solid foundation.

However, this review identifies 15 issues across four categories that affect the experience for real users. The most impactful problems are:

1. **Hardcoded English throughout** — the About, Support, and Download pages, AI settings panel, topics tab labels, verse-detail section headers, phrase list, and numerous inline strings bypass the i18n system entirely, making the app effectively English-only for these features.
2. **Inconsistent loading/empty states** — some pages have excellent skeleton loaders and error display, while newer features (topics AI tab, phrase list, tafsir, discussion) use raw "Loading..." text without i18n or skeleton structure.
3. **Accessibility gaps in interactive widgets** — AI toggle buttons, word-analysis cards, settings checkboxes, and topic category headers use custom elements without sufficient ARIA roles or keyboard navigation patterns.
4. **Footer link to defunct Twitter/X account** — the footer links to `twitter.com/ProjThaqalayn` which may no longer be active or relevant.

None of these are critical blockers (Phase 1 addressed those), but they collectively undermine the polish of what is otherwise a well-built scripture reading application.

## Desktop Findings

### HIGH Severity

**D-01: About and Support Pages Entirely Hardcoded in English**

Both `/about` and `/support` pages contain zero i18n `translate` pipe calls. Every heading, paragraph, and link text is hardcoded English. This is the largest i18n gap in the application — two entire pages that non-English users cannot read.

- Files: `about.component.html`, `support.component.html`
- Impact: Users in 11 of 12 supported languages see untranslated content

**D-02: AI Settings Panel Labels Hardcoded in English**

The settings component's AI preferences panel (`settings.component.html` lines 28-72) has 7 hardcoded English strings: "AI Feature Settings", "Show diacritized text by default", "Show content type badges", "Show topic tags", "Show isnad/matn separation", "Show AI translation disclaimer", "Word-by-word language:". None use the `translate` pipe.

- File: `settings.component.html`
- Impact: AI feature toggles are unusable for non-English readers

**D-03: Verse-Detail Section Headers Hardcoded in English**

In `verse-detail.component.html`, three section headings bypass i18n: "Quran References" (line 134), "Key Phrases" (line 150), "Related Narrations" (line 164). These should use `translate` pipe like the rest of the component does.

- File: `verse-detail.component.html`

**D-04: Topics Page Tab Labels and Content Hardcoded in English**

The topics component has hardcoded tab labels "Books", "AI Topics", "Key Phrases" (lines 8-15), AI topic counts use "hadiths" without i18n (lines 80, 89), and the "coming soon" messages and "View Key Phrases" link text are all English-only.

- File: `topics.component.html`

**D-05: Phrase List Page Entirely Hardcoded in English**

The phrase-list component has zero `translate` pipe usage. "Key Phrases", "Common Islamic multi-word expressions...", "Loading phrases...", "Search phrases...", "No phrases match your search", "Key phrase index is coming soon..." are all hardcoded.

- File: `phrase-list.component.html`

### MEDIUM Severity

**D-06: Download Page Has Hardcoded Strings Mixed with Translated Ones**

The download page partially uses i18n but has hardcoded "Exporting..." and "Export CSV" (line 69). The open source section text (lines 77-95) is entirely hardcoded English.

- File: `download.component.html`

**D-07: Tafsir Loading Text Hardcoded**

In chapter-content, the tafsir loading state says "Loading commentary..." (line 236) in hardcoded English instead of using the translate pipe.

- File: `chapter-content.component.html`

**D-08: Verse-Text AI Toggle Aria-Labels Hardcoded**

The verse-text component has hardcoded English aria-labels on AI toggle buttons: "Show original text", "Show diacritized text", "Show narrator chain diagram", "Hide word analysis", etc. (lines 6-27). While `matTooltip` uses translate pipe, the `aria-label` attributes do not.

- File: `verse-text.component.html`

**D-09: Book-Dispatcher Default Case Shows Raw English Error**

When `book.kind` doesn't match any known type, the component shows "Not sure how to display this book part, sorry!" (line 57) — a raw English string with no i18n key and no error-display component.

- File: `book-dispatcher.component.html`

**D-10: Search Results Filter Banner Has Hardcoded English**

The search results component has a filter banner with hardcoded "Filtered by" text (line 37) instead of using the translate pipe.

- File: `search-results.component.html`

### LOW Severity

**D-11: Footer Links to Twitter Account (Potentially Defunct)**

The footer contains a link to `https://twitter.com/ProjThaqalayn` labeled with `'nav.news' | translate`. Twitter has rebranded to X and the account may be inactive. This could confuse users or lead to a dead link.

- File: `app.component.html` line 112

**D-12: Inconsistent Card Elevation Pattern**

Some content containers use `class="content mat-elevation-z8"` (about, support, download, bookmarks, topics, phrase-list) while others use `mat-card appearance="outlined"`. The z8 elevation creates a heavy shadow that doesn't match the subtle card design used in verse display.

- Files: Multiple component templates

## Mobile Findings

### HIGH Severity

**M-01: AI Toggle Buttons Too Small for Touch on Mobile**

The AI toggle buttons in verse-text have `padding: 2px` with 16x16px icons (verse-text.component.scss lines 89-114). On mobile, these are approximately 20px total touch area — well below the 44px minimum. Unlike the verse footer actions (which were fixed in FIX-05), these AI toggles were not included in the touch target fix.

- File: `verse-text.component.scss`
- Impact: Users cannot reliably tap diacritics, word analysis, or paragraph view toggles on mobile

**M-02: Word-Analysis Popup Positioning May Overflow Viewport on Mobile**

The word detail popup uses absolute positioning with `left` and `top` in pixels (`[style.left.px]="wordPopup.x"`, `[style.top.px]="wordPopup.y"`). On narrow mobile screens, the popup (min-width: 160px) can extend beyond the viewport edge. There is no viewport boundary clamping logic visible in the template or styles.

- File: `verse-text.component.html` lines 80-96, `verse-text.component.scss` lines 275-346

### MEDIUM Severity

**M-03: Mobile Menu Lacks AI Settings**

The mobile hamburger menu (app.component.html lines 53-104) includes navigation links, dark mode, font size, and language picker, but does not include AI feature preferences. Users on mobile must find and use the settings component embedded in chapter-content to toggle AI features. The acceptance criteria for MOB-01 specified "ALL settings consolidated" including "AI content preferences."

- File: `app.component.html`

**M-04: Bottom Navigation Missing Narrators Link**

The bottom navigation bar has 4 items: Home, Books, Topics, Bookmarks. The Narrators section (`/people/narrators`) — a major feature — is only accessible from the hamburger menu or footer. Given that the hamburger menu is a secondary action and the footer is hidden on mobile (replaced by bottom nav), Narrators discovery is poor on mobile.

- File: `app.component.html` lines 119-136

**M-05: Compact Header on Scroll Reduces Breadcrumb Legibility**

When the compact header activates on scroll (at 414px width), breadcrumb font size drops to 9px (app.component.scss line 436). At this size, breadcrumb text — especially Arabic titles — becomes illegible on high-DPI mobile screens.

- File: `app.component.scss` lines 424-438

### LOW Severity

**M-06: Tafsir Edition Select is Native HTML Select, Not Material**

The tafsir edition selector in chapter-content uses a plain `<select>` element instead of `<mat-select>`. This creates an inconsistency with the rest of the UI which uses Material form fields. On mobile, native selects behave differently (OS-level picker vs Material dropdown).

- File: `chapter-content.component.html` line 227-231

## Accessibility Findings

### HIGH Severity

**A-01: Topic Category Headers Used as Buttons Without Role**

In the topics component, `<h2>` elements have `(click)` handlers for expanding/collapsing AI topic categories (line 77-81) but lack `role="button"`, `tabindex="0"`, or keyboard event handlers (`keydown.enter`/`keydown.space`). Screen reader and keyboard users cannot interact with these.

- File: `topics.component.html`

**A-02: AI Settings Checkboxes Use Native Inputs Without Labels**

The AI settings panel uses `<label>` elements wrapping `<input type="checkbox">` and `<span>` elements (settings.component.html lines 33-73), which is technically accessible. However, the labels are hardcoded English (see D-02), and the checkboxes lack `id`/`for` pairing — relying on implicit label association which some screen readers handle poorly.

- File: `settings.component.html`

### MEDIUM Severity

**A-03: Discussion Section Toggle Lacks ARIA Attributes**

The discussion header in verse-detail uses `(click)="showCommentEditor = !showCommentEditor"` on a `<div>` (line 247). This div acts as a button but has no `role="button"`, `tabindex`, or ARIA expanded state.

- File: `verse-detail.component.html`

**A-04: Embed Verse Component Lacks Lang Attributes on Translation Text**

The embed-verse component marks Arabic text with `lang="ar" dir="rtl"` correctly, but translation text paragraphs (lines 23-27, 55-59) have no `lang` attribute. The translation language varies, so the correct `lang` value should be set dynamically based on the selected translation.

- File: `embed-verse.component.html`

**A-05: Search Dropdown Results Missing ARIA Selected State**

The search dropdown items have `role="option"` but the container uses `role="listbox"` without `aria-activedescendant` or arrow-key navigation support. Users cannot navigate results with keyboard alone.

- File: `search-bar.component.html`

### LOW Severity

**A-06: Narrator Table Sort Headers Missing Accessible Names**

This is an existing known issue (tracked in `KNOWN_ISSUE_RULES_TO_SKIP` in accessibility.spec.ts as `aria-command-name`). Material table sort headers lack accessible names. Noting for completeness — this remains unresolved.

## Recommendations for Phase 13

Based on severity, user impact, and effort, the top priorities for Phase 13 are:

1. **i18n completion** (D-01 through D-05, D-06, D-07) — the single largest gap. Approximately 50+ hardcoded strings across 7 components need i18n keys added to all 12 language files.
2. **AI toggle touch targets on mobile** (M-01) — direct accessibility/usability blocker.
3. **Interactive element accessibility** (A-01, A-03, A-05) — keyboard and screen reader users blocked from features.
4. **Mobile AI settings gap** (M-03) — MOB-01 acceptance criteria not fully met.
5. **Word popup viewport clamping** (M-02) — prevents content from being cut off on mobile.

Lower priority but valuable:
- Bottom nav narrators link (M-04)
- Consistent elevation patterns (D-12)
- Footer social link audit (D-11)
- Embed verse lang attributes (A-04)
- Tafsir select Material consistency (M-06)
